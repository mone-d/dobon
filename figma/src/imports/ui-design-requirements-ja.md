# UI デザイン要件 - ドボンカードゲーム

## プロジェクト概要
2〜8人のプレイヤーがオンラインでリアルタイムに「ドボン」をプレイするモバイルカードゲームアプリケーション。素早い意思決定と全てのゲーム要素の明確な可視性が求められる。

---

## ターゲットユーザーと利用状況

### 主要ユーザー
- カジュアルにプレイするチームメンバー
- 年齢層: 大人（20〜40代）
- 技術習熟度: 基本的なスマートフォンユーザー

### 利用状況
- **主要デバイス**: スマートフォン
- **主要向き**: 横画面（ランドスケープ）
- **セッション時間**: 1ゲーム10〜30分
- **環境**: カジュアルな場所（自宅、休憩時間）
- **ネットワーク**: WiFiまたは4G/5Gモバイルデータ

### ユーザーの目標
1. 現在のゲーム状態を素早く理解する
2. 全てのカード（自分と相手）を簡単に識別する
3. 素早く決断する（カードを出す、ドボンを宣言する）
4. ゲームの進行とスコアを追跡する
5. スムーズで反応の良いゲームプレイを楽しむ

## コアデザイン要件

### 1. 情報の優先順位

**最重要情報（常に表示）**:
- 自分の手札 - 全てのカードを明確に見る必要がある
- 場札 - 現在プレイ中のカード
- 相手のカード枚数 - 各プレイヤーが何枚持っているか
- 現在のターン表示 - 誰のターンか
- ドボンボタン - 素早い宣言のため常にアクセス可能

**重要情報（関連時に表示）**:
- 相手の公開カード - 公開されたカード
- 山札の残り枚数 - 残りカード数
- 倍率 - 現在の支払い倍率
- ドボン表示 - ドボン発生時、誰が何のカードで宣言したか

**二次情報**:
- プレイヤー名
- 最後にカードを出した人
- ゲームフェーズ表示

### 2. 画面領域の制約

**課題**: スクロールなしで全てを1画面に収める
- 最大8人のプレイヤー
- 各プレイヤーは最大13枚のカードを持つ
- 自分の手札も最大13枚
- ドボン表示（発生時）
- 操作ボタン

**要件**:
- メインゲームプレイで横スクロールなし
- 相手リストのみ縦スクロール可（必要な場合）
- 全ての重要情報が一度に見える
- 効率的なスペース使用

### 3. カード表示要件

**自分の手札**:
- 全てのカードが識別可能である必要がある
- 多数のカード（6枚以上）の場合、スペース効率の良い表示を使用
- 各カードはスート（♠♥♦♣）と値（A〜K）を表示
- 選択されたカードは明確に区別される
- 公開カードと非公開カードは視覚的に異なる

**相手のカード**:
- カード枚数が目立つ必要がある
- 公開カードは見える（スートと値）
- 非公開カードはカード裏面として表示
- 13枚全て公開の場合、全て識別可能である必要がある
- 複数プレイヤーを収めるためのコンパクト表示

**場札**:
- 大きく目立つ
- スートと値が見やすい
- 誰が出したか明確に表示

### 4. インタラクション要件

**カード操作**:
- タップで選択/選択解除
- ドラッグ&ドロップでカードをプレイ
- 全てのインタラクションに視覚的フィードバック
- 異なる値を選択時に自動選択解除（ビジネスルール）

**ドボン宣言**:
- ワンタップで宣言（確認不要）
- 常にアクセス可能なボタン
- システムが全ての計算式を自動計算
- 宣言時に明確なフィードバック

**カードを引く**:
- 山札をタップして引く（自分のターン時）
- 引けるタイミングの明確な表示
- 引く動作の視覚的フィードバック

### 5. リアルタイム更新

**ゲーム状態の変化**:
- プレイヤーが行動した際の即座の更新
- カード移動のスムーズなアニメーション
- ターン変更の明確な表示
- ドボン発生時の視覚的インパクト

**マルチプレイヤー認識**:
- 全プレイヤーの状態を同時に見る
- 現在のターンプレイヤーを明確に表示
- 全員にドボン宣言が見える
- 返しドボンチェーンの可視化

## 機能要件

### レイアウト構造

**全体構成**:
- 画面を論理的なゾーンに分割
- 相手エリア（コンパクト、必要に応じてスクロール可能）
- ゲームフィールドエリア（山札、場札、ドボン表示）
- プレイヤー手札エリア（下部、常に表示）
- フローティングコントロール（ドボンボタン）

**レスポンシブ動作**:
- スマートフォン横画面に最適化
- 縦画面への適切な対応（ユーザーが回転した場合）
- 異なる画面サイズへの対応（5〜7インチスマホ）

### カード表示ロジック

**通常状態（少数のカード）**:
- 快適な間隔でカードを表示
- 完全なカード詳細が見える
- タップ/選択が簡単

**コンパクト状態（多数のカード）**:
- スペース節約のためカードを重ねる
- スートと値は見えるようにする
- 任意のカードを選択できる能力を維持
- ホバー/タップで完全なカードを表示

**相手のカード**:
- カード枚数の可視性を優先
- 公開カードをコンパクトに表示
- 多数のカードの場合はスタックまたは重ねる
- 現在のターンプレイヤーを区別

### ドボン表示

**ドボン発生時**:
- ドボン宣言を目立つように表示
- 表示内容: 誰が宣言、使用カード、計算式、結果
- 重要なゲーム要素を隠さない
- コンパクトだが読みやすい

**返しドボンチェーン**:
- 複数のドボンを視覚的にスタック
- シーケンスを明確に表示
- 誰が誰を返したかを示す
- コンパクトなフットプリントを維持

### 視覚的フィードバック

**インタラクティブ要素**:
- 全てのクリック可能要素にホバー状態
- ボタンのアクティブ/押下状態
- カードドラッグのドラッグプレビュー
- ドロップターゲットの表示

**状態変化**:
- 状態間のスムーズな遷移
- カード移動のアニメーション
- 重要イベント（ドボン）の注目を集める表現
- 日常的な動作（ターン変更）の控えめな表現

### 明確性
- ゲーム状態を即座に理解
- 誰のターンか混乱しない
- 利用可能なアクションの明確な表示
- 全てのインタラクションに明白なフィードバック

### 効率性

- 一般的な操作への素早いアクセス

### 応答性

- スムーズなアニメーション（60fps）
- 遅延やラグなし
- リアルタイム更新が即座に感じられる

### 美観
- カードゲームの雰囲気
- プロフェッショナルだが遊び心がある
- 情報密度が高いにもかかわらず雑然としていない
- 視覚的に魅力的な配色

## 解決すべきデザイン課題

### 課題1: 13枚の手札表示
**問題**: フルサイズの13枚のカードは画面幅に収まらない
**要件**:
- 13枚全てのカードが識別可能である必要がある
- 各カードのスートと値が見える必要がある
- ユーザーは任意のカードを選択できる
- 横スクロールなし

**望ましい解決策**:
- 創造的な重ねまたはスタック
- スペースの賢い使用
- 使いやすさの維持

### 課題2: 8プレイヤー + カード枚数
**問題**: 様々なカード枚数を持つ8人のプレイヤーをコンパクトに表示する必要がある
**要件**:
- 全プレイヤーが同時に見える
- カード枚数が目立つ
- 公開カードが見える
- 現在のターンプレイヤーが明白

**望ましい解決策**:
- 効率的なリストレイアウト
- 視覚的階層
- コンパクトなカード表現

### 課題3: ブロックしないドボン表示
**問題**: ドボン表示はゲームプレイをブロックせずに情報を表示する必要がある
**要件**:
- ドボン詳細を表示（プレイヤー、カード、計算式）
- 場札やプレイヤーの手札を隠さない
- 複数のドボン（返しチェーン）をサポート
- コンパクトだが読みやすい

**望ましい解決策**:
- スマートな配置
- コンパクトなカード表示
- 複数ドボンのスタック

### 課題4: 1画面収め
**問題**: 全ての要素をスクロールなしで収める必要がある
**要件**:
- 相手リスト
- ゲームフィールド（山札、場札、ドボン）
- プレイヤーの手札（最大13枚）
- 操作ボタン
- 全て同時に見える

**望ましい解決策**:
- 効率的なスペース配分
- 相手リストのみ縦スクロール
- スマートなレスポンシブレイアウト

## インタラクションパターン

### カード選択
- **アクション**: 手札のカードをタップ
- **フィードバック**: 視覚的ハイライト（枠、浮き上がり、または光）
- **ルール**: 異なる値の場合は前の選択を自動解除
- **状態**: 選択されたカードはハイライトされたまま

### カードをプレイ
- **アクション**: カードを場にドラッグ または 選択したカードをタップして場をタップ
- **フィードバック**: ドラッグプレビュー、ドロップターゲットハイライト
- **検証**: 無効な場合、手札に戻る（エラーメッセージなし）
- **結果**: カードが場に移動、ターンが進む

### ドボン宣言
- **アクション**: ドボンボタンをタップ
- **フィードバック**: ボタン押下アニメーション
- **処理**: システムが全ての計算式を計算
- **結果**: ドボン表示が現れる、または無効な場合はペナルティ

### カードを引く
- **アクション**: 山札をタップ（利用可能な時）
- **フィードバック**: 山札ハイライト、引くアニメーション
- **結果**: カードが手札に追加される

### エラー防止

- 有効なアクションの明確な表示

### パフォーマンス
- 多数の要素があってもスムーズなアニメーション
- インタラクション中の遅延なし

## デザイン制約

### 技術的制約
- Webアプリケーション（ネイティブアプリではない）
- 最新のモバイルブラウザで動作する必要がある
- タッチ最適化（マウスホバー不要）
- WebSocketによるリアルタイム更新

### ビジネス制約
- POCレベル（本番環境レベルではない）
- チーム使用のみ（一般公開なし）
- アクセシビリティ要件なし（WCAG）
- 開発期間: 1〜2週間

### デバイス制約
- スマートフォン画面（5〜7インチ）
- 横画面が主
- 縦画面はフォールバック
- 最新デバイス（過去3年）

## デザイナーへの注意事項

- これはリアルタイムマルチプレイヤーカードゲームです
- スピードと明確性が装飾よりも重要です
- 情報密度が高い - スマートなスペース使用が重要です
- モバイルファースト、タッチ最適化デザインが必要です
- 13枚カード表示の課題が最も難しい問題です
- ユーザーは素早い決断をする必要がある - 明確性が最優先です
- カードゲームの雰囲気だがモダン/デジタルな感じ
- POCレベルだがプロフェッショナルに見える必要があります

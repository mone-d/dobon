# NFR Design Plan - Unit 1 (Frontend)

## ユニット概要

**Unit 1: フロントエンド**
- 責務: ゲーム画面、ロビー・ルーム管理、プロフィール・統計画面の実装
- 技術スタック: Vue.js 3 + TypeScript + Pinia
- 実装方針: モックデータ、モックAPI、モックWebSocket を使用

---

## NFR Design 計画

### [ ] 1. レジリエンス（復帰）パターン

#### 1.1 WebSocket 接続切断時の復帰戦略

NFR 要件では「接続切断時の復帰は 5 秒以内」と定義されています。

WebSocket 接続切断時の復帰メカニズムについて、以下の質問にお答えください：

**Q1.1.1 自動再接続の実装方法**

Socket.io の自動再接続機能を使用する場合、以下のどちらを選択しますか？

A. Socket.io のデフォルト再接続（推奨）
   - 自動的に再接続を試みる
   - リトライ間隔: 1秒、2秒、3秒、4秒、5秒
   - 最大リトライ回数: 5 回

B. カスタム再接続ロジック
   - より細かい制御が必要
   - 独自のリトライ戦略を実装

[Answer]:A

---

**Q1.1.2 接続切断時のユーザーへの通知**

接続が切れた場合、ユーザーにどのように通知しますか？

A. 画面上部にバナーを表示
   - 「接続が切れました。再接続中...」
   - 再接続成功時に自動的に消える

B. モーダルダイアログを表示
   - より目立つ通知
   - ユーザーが確認する必要がある

C. 通知なし
   - バックグラウンドで再接続
   - ユーザーには見えない

[Answer]:A

---

**Q1.1.3 再接続失敗時の処理**

5 回のリトライすべてが失敗した場合、どのように処理しますか？

A. ユーザーに手動再接続ボタンを表示
   - ユーザーが「再接続」ボタンをクリック
   - 再度接続を試みる

B. ページをリロードするよう促す
   - 「ページをリロードしてください」というメッセージ

C. ゲーム画面から退出
   - ロビー画面に戻す

[Answer]:B

---

#### 1.2 API エラー時のリトライ戦略

REST API（Axios）のエラー時のリトライについて、以下の質問にお答えください：

**Q1.2.1 API エラー時のリトライ回数**

API リクエストが失敗した場合、何回までリトライしますか？

A. 1 回（リトライなし）
   - エラーが発生したら即座にユーザーに通知

B. 3 回
   - 一時的なエラーに対応

C. 5 回
   - より多くの再試行

[Answer]:B

---

**Q1.2.2 リトライ間隔**

API リトライの間隔は？

A. 固定間隔（例：1 秒ごと）
   - シンプルな実装

B. 指数バックオフ（例：1秒、2秒、4秒）
   - サーバー負荷を軽減

C. ランダム間隔
   - より分散した再試行

[Answer]:A

### [ ] 2. スケーラビリティパターン

#### 2.1 状態管理のスケーリング

NFR 要件では「同時接続 10 人、月間 100 ゲーム」と定義されています。

Pinia ストアのスケーリングについて、以下の質問にお答えください：

**Q2.1.1 ゲーム履歴の保存方法**

ゲーム履歴（月間 100 ゲーム）をどこに保存しますか？

A. ローカルストレージのみ
   - ブラウザのローカルストレージに保存
   - 容量制限あり（通常 5-10MB）

B. ローカルストレージ + IndexedDB
   - より大容量のデータを保存
   - 複雑な検索が可能

C. サーバーのみ
   - バックエンド API に保存
   - ローカルストレージは使用しない

[Answer]:C

---

**Q2.1.2 メモリ内ストアのサイズ管理**

Pinia ストアに保持するゲーム履歴の最大件数は？

A. 最新 10 ゲーム
   - メモリ効率重視

B. 最新 50 ゲーム
   - バランス型

C. すべてのゲーム（100 ゲーム）
   - 完全な履歴を保持

[Answer]:A

---

#### 2.2 コンポーネント数のスケーリング

**Q2.2.1 コンポーネント分割戦略**

   - プレゼンテーション層、ロジック層等

C. 両方を組み合わせ

[Answer]:A

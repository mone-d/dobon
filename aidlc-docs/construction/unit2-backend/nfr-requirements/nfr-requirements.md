# NFR Requirements - Unit 2 (Backend - Game Logic)

## 概要

Unit 2（バックエンド - ゲームロジック）の非機能要件を定義する。
POCレベルのプロジェクトであり、チームメイトのみが利用するため、セキュリティと品質要件は最小限に設定する。

---

## 1. スケーラビリティ要件

### 1.1 同時ゲームセッション数

**要件**: 同時ゲームセッション数は **1セッション** を想定

**設計方針**:
- チームメイト全員で1ゲームを想定
- メモリ上の Map でセッション管理（スケーリング不要）
- 複数セッション対応のコードは書くが、最適化は不要

### 1.2 同時接続プレイヤー数

**要件**: Unit 1 と同様、**10人程度** を想定

**設計方針**:
- Socket.io の接続管理で十分対応可能
- 大規模スケーリング（クラスタリング、Redis Adapter）は不要

---

## 2. パフォーマンス要件

### 2.1 ゲームロジック処理時間

**要件**: 特に目標なし（POCレベル）

**設計方針**:
- ゲームロジックはメモリ内処理のため、実用上問題ない速度で動作する
- ドボン演算式計算（4演算子 × 手札枚数）は最大でも数ms以内
- ボトルネックが発生した場合のみ対処

### 2.2 WebSocket イベント配信

**要件**: イベント配信遅延は **ネットワーク遅延のみ**（サーバー処理は無視できる程度）

**設計方針**:
- Socket.io の emit は同期的に処理
- ブロッキング処理を避ける（非同期処理を徹底）

---

## 3. 可用性・復帰要件

### 3.1 サーバー再起動時の扱い

**要件**: 再起動時にゲームセッションは **消える**（POCレベルで許容）

**設計方針**:
- メモリのみで管理するため、再起動でセッションはリセット
- 永続化（ファイル・DB）は不要
- ユーザーはルームに戻って再度ゲームを開始する

### 3.2 クライアント再接続時の対応

**要件**: クライアントが再接続した場合、**ゲーム状態を再送信**する

**設計方針**:
- Socket.io の `reconnect` イベントを検知
- 再接続時に現在の GameState をクライアントに送信
- セッションIDで再接続プレイヤーを識別

---

## 4. セキュリティ要件

### 4.1 API 認証

**要件**: 認証なし（POCレベル、チームメイトのみ）

**設計方針**:
- JWT・セッション認証は不要
- ゲストIDをプレイヤー識別子として使用
- 信頼ベースの運用

### 4.2 入力バリデーション

**要件**: **最低限の検証**（必須パラメータの存在確認のみ）

**設計方針**:
- 必須フィールドの null/undefined チェック
- 型チェックは TypeScript の型定義で対応
- 詳細な範囲チェック・サニタイズは不要

### 4.3 通信の暗号化

**要件**: 本番環境では **HTTPS/WSS** を使用

**設計方針**:
- 開発環境: HTTP/WS（ポート3000）
- 本番環境: Heroku が自動でHTTPS/WSSを提供
- SSL証明書の手動管理は不要

### 4.4 CORS 設定

**要件**: フロントエンド（Vercel）からのアクセスを許可

**設計方針**:
- 開発環境: `*`（全オリジン許可）
- 本番環境: Vercel のドメインのみ許可
- Socket.io の CORS 設定で管理

---

## 5. 信頼性・エラーハンドリング要件

### 5.1 ゲーム状態の不整合

**要件**: 不整合検出時は **例外をスローしてゲームを終了**

**設計方針**:
- try-catch でグローバルエラーハンドリング
- エラー発生時は全プレイヤーに `game:error` イベントを送信
- ゲームセッションをメモリから削除
- 詳細なエラーログをコンソールに出力

### 5.2 タイムアウト処理

**要件**: 返しドボン判定は **10秒でタイムアウト**

**設計方針**:
- `setTimeout` で各セッションごとに管理（シンプルな実装）
- タイムアウト時は pending プレイヤーを「返さない」として処理
- セッション終了時に `clearTimeout` でクリーンアップ

---

## 6. ログ要件

### 6.1 ログ出力レベル

**要件**: **詳細ログ**（全ゲームイベント、デバッグ情報、エラー）

**設計方針**:
- 全ゲームイベントをログ出力（カード操作、ドボン宣言、支払い等）
- エラーは詳細なスタックトレース付き
- デバッグ情報（ゲーム状態の変化）もログ出力

### 6.2 ログ保存先

**要件**: **コンソール出力のみ**

**設計方針**:
- `console.log` / `console.error` でコンソールに出力
- Heroku ダッシュボードの「Logs」タブから確認可能
- ファイル出力・外部ログサービスは不要（POCレベル）
- ログライブラリ（winston等）は不要、シンプルなコンソール出力で十分

---

## 7. テスト要件

### 7.1 テストフレームワーク

**要件**: **Jest** を使用

**設計方針**:
- Node.js バックエンドの標準的なテストフレームワーク
- TypeScript 対応（ts-jest）
- カバレッジ目標なし（POCレベル）

### 7.2 テスト対象の優先度

**要件**: **ゲームロジック全体** を優先してテスト

**テスト対象**（優先度順）:
1. `DoboDeclaration` - 演算式自動計算（最も複雑）
2. `MultiplierCalculator` - 倍率計算の全パターン
3. `CardValidator` - カード操作ルール検証
4. `PaymentCalculator` - 支払い計算
5. `DeckManager` - デッキ初期化・再生成
6. `GameEngine` - ゲームフロー（統合的）

### 7.3 テストデータ

**要件**: ファクトリーパターンでテストデータを生成（Functional Design で決定済み）

---

## 8. デプロイ要件

### 8.1 ホスティング

**要件**: **Heroku** を使用

**設計方針**:
- Heroku の無料枠（Eco Dynos）を使用
- WebSocket 対応済み
- GitHub 連携で自動デプロイ
- HTTPS/WSS は Heroku が自動提供

### 8.2 環境設定

**開発環境**:
- ポート: 3000（`process.env.PORT` で上書き可能）
- HTTP/WS

**本番環境（Heroku）**:
- ポート: Heroku が自動割り当て（`process.env.PORT`）
- HTTPS/WSS（Heroku 自動）

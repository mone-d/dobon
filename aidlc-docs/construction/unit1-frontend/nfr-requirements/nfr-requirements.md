# NFR Requirements - Unit 1 (Frontend)

## 概要

Unit 1（フロントエンド）の非機能要件（NFR）を定義します。POC レベルのプロジェクトであり、チームメイトのみが利用するため、セキュリティと品質要件は最小限に設定されています。

---

## 1. スケーラビリティ要件

### 1.1 同時接続プレイヤー数

**要件**: 同時接続プレイヤー数は **10人程度** を想定

**設計方針**:
- チームメイトのみの利用を想定
- 小規模な同時接続数に最適化
- グローバルストア（Pinia/Redux）で十分
- 大規模スケーリングは不要

### 1.2 ゲーム数・データ成長

**要件**: 1ヶ月のゲーム数は **100ゲーム程度** を想定

**設計方針**:
- ゲーム履歴はローカルストレージに保存
- 月間 100 ゲーム程度のデータ量は問題なし
- データベース設計は簡潔に
- アーカイブ機能は不要

---

## 2. パフォーマンス要件

### 2.1 ドボン宣言のレスポンスタイム

**要件**: ドボン宣言から結果表示までは **500ms 以内**

**設計方針**:
- モック API で即座に応答
- WebSocket で返し判定フェーズに移行
- ネットワーク遅延を考慮した設計
- 実装時は API レスポンスを 200-300ms に設定

### 2.2 カード操作のレスポンスタイム

**要件**: カード操作（ドラッグ&ドロップ）は **50ms 以内**

**設計方針**:
- UI 更新は即座に反応
- ドラッグ中のビジュアルフィードバック必須
- 60fps アニメーション対応
- requestAnimationFrame を活用

### 2.3 初期ロード時間

**要件**: ゲーム画面の初期ロード時間は **3秒以内**

**設計方針**:
- バンドルサイズ最適化（コード分割）
- 画像の遅延読み込み
- キャッシング戦略の実装
- 開発環境では 1-2 秒を目標

---

## 3. 可用性・復帰要件

### 3.1 接続切断時の復帰

**要件**: ゲーム中に接続が切れた場合、**5秒以内** に復帰可能

**設計方針**:
- WebSocket 自動再接続機能
- 最大 5 回まで自動リトライ
- リトライ間隔: 1秒、2秒、3秒、4秒、5秒
- 5 回失敗後はユーザーに通知

### 3.2 オフライン対応

**要件**: オフライン対応は **不要**

**設計方針**:
- POC レベルのため、オフライン機能は実装しない
- 接続が必須の前提で設計
- 接続切断時はエラーメッセージを表示

---

## 4. セキュリティ要件

### 4.1 ゲストID の保護

**要件**: ゲストID の保護は **最小限**

**設計方針**:
- ゲストID はプレーンテキストで保存
- ローカルストレージに保存
- ユーザーが手動で入力・削除可能
- 暗号化は不要

### 4.2 ゲーム結果の改ざん防止

**要件**: ゲーム結果の改ざん防止は **不要**

**設計方針**:
- チームメイトのみの利用を信頼ベースで想定
- チェックサムやハッシュ検証は不要
- ゲーム結果はそのまま保存

### 4.3 通信の暗号化

**要件**: HTTPS/WSS による通信暗号化は **必須**

**設計方針**:
- 本番環境では HTTPS/WSS を使用
- 開発環境では HTTP/WS でも可
- SSL/TLS 証明書の設定が必要
- ブラウザの混合コンテンツ警告に対応

---

## 5. ブラウザ互換性

### 5.1 対応ブラウザ

**要件**: 対応ブラウザは **最新版のみ**

**対応ブラウザ**:
- Chrome 最新版
- Safari 最新版
- Firefox 最新版
- Edge 最新版

**設計方針**:
- ES2020 以上の JavaScript 機能を使用可
- ポリフィルは不要
- 古いブラウザ対応は不要

### 5.2 モバイルブラウザ対応

**要件**: iOS Safari・Android Chrome 対応は **必須**

**設計方針**:
- iOS Safari 最新版対応
- Android Chrome 最新版対応
- タッチイベント対応
- モバイル特有の問題に対応

### 5.3 画面サイズ対応

**要件**: 対応画面サイズは **スマートフォンのみ**（375px～）

**設計方針**:
- 最小幅: 375px（iPhone SE）
- 最大幅: 812px（iPhone 12 Pro Max）
- タブレット・デスクトップ対応は不要
- レスポンシブデザイン: モバイルファースト

---

## 6. テスト・品質要件

### 6.1 ユニットテストカバレッジ

**要件**: ユニットテストカバレッジは **目標なし**（適当）

**設計方針**:
- POC レベルのため、テストは最小限
- 主要なビジネスロジックのみテスト
- UI コンポーネントのテストは不要
- テストカバレッジ: 30-50% 程度を想定

### 6.2 E2E テスト

**要件**: E2E テストは **不要**

**設計方針**:
- POC レベルのため、E2E テストは実装しない
- 手動テストで十分
- 主要フローの動作確認のみ

### 6.3 コード品質ツール

**要件**: ESLint、Prettier、TypeScript strict mode は **不要**

**設計方針**:
- POC レベルのため、コード品質ツールは最小限
- TypeScript は使用（型安全性のため）
- ESLint・Prettier は不要
- コード品質は開発者の裁量に任せる

---

## 7. ログ・監視要件

### 7.1 ログ詳細度

**要件**: ゲーム中のエラーログやデバッグ情報は **詳細**

**設計方針**:
- すべてのゲームイベントをログ出力
- ドボン宣言、返し判定、支払い等をログ
- エラーは詳細なスタックトレース付き
- ブラウザコンソールに出力
- 本番環境では console.log を削除

### 7.2 本番環境での監視

**要件**: 本番環境での監視・アラートは **不要**

**設計方針**:
- POC レベルのため、監視機能は不要
- エラーが発生した場合はユーザーに通知
- ユーザーが手動で報告する仕組み

---

## 8. アクセシビリティ要件

### 8.1 WCAG 対応

**要件**: WCAG 対応は **不要**

**設計方針**:
- POC レベル、チームメイトのみの利用
- アクセシビリティ対応は不要
- 基本的な UI 操作のみ実装

### 8.2 キーボード操作

**要件**: キーボード操作のサポートは **不要**

**設計方針**:
- マウス/タッチ操作のみ対応
- キーボード操作は実装しない
- ゲームの性質上、マウス/タッチが主体

---

## 9. 技術スタック選定

### 9.1 フロントエンドフレームワーク

**選定**: Vue.js 3 または React

**理由**:
- ユーザーが Vue.js に親しんでいる
- React も選択肢として可能
- 両者とも最新版を使用

### 9.2 状態管理

**選定**: Pinia（Vue.js）または Redux（React）

**理由**:
- 小規模プロジェクトに適切
- グローバル状態管理が必要
- 学習コストが低い

### 9.3 UI コンポーネントライブラリ

**選定**: 不要（カスタム実装）

**理由**:
- POC レベルのため、シンプルな UI で十分
- カード、ボタン等は自作
- 外部ライブラリの依存を最小化

### 9.4 通信ライブラリ

**選定**: Axios（REST API）、Socket.io（WebSocket）

**理由**:
- Axios: REST API 通信の標準
- Socket.io: WebSocket の簡単な実装
- 両者とも実績が豊富

### 9.5 ビルドツール

**選定**: Vite

**理由**:
- 高速なビルド・開発サーバー
- Vue.js 3 との相性が良い
- 最新の標準ツール

---

## 10. 実装上の注意点

### 10.1 パフォーマンス最適化

- コード分割による初期ロード時間の短縮
- 画像の遅延読み込み
- 不要な再レンダリングの回避（メモ化）
- 仮想スクロール（リスト表示時）

### 10.2 エラーハンドリング

- API エラー時のリトライ機能
- WebSocket 接続切断時の自動再接続
- ユーザーへの分かりやすいエラーメッセージ

### 10.3 デバッグ・開発効率

- Vue DevTools / React DevTools の活用
- ブラウザコンソールでの詳細ログ出力
- ホットリロード機能の活用

### 10.4 本番環境への対応

- HTTPS/WSS の設定
- 環境変数による API エンドポイント切り替え
- console.log の削除またはフィルタリング
- ソースマップの生成（デバッグ用）

---

- [ ] キーボード操作サポート
- [ ] E2E テスト
- [ ] ESLint・Prettier 等のコード品質ツール
- [ ] 本番環境での監視・アラート
- [ ] オフライン対応
- [ ] ゲーム結果の改ざん防止
- [ ] 大規模スケーリング対応

# Business Logic Model - Unit 1 (Frontend)

## フロントエンドのビジネスロジック

### 1. ゲーム画面のロジック

#### 1.1 カード操作ロジック

**ドラッグ操作でカードを出す**

```
1. ユーザーが手札のカードをタップして選択
   - 選択されたカードがハイライト表示
   - 複数枚選択可能（同じ数字のみ）

2. ユーザーが異なる数字のカードをタップ
   - 前の選択が自動的に解除される
   - 新しく選択されたカードがハイライト表示

3. ユーザーが選択したカードをドラッグ
   - ドラッグ中はカードが半透明表示
   - ドラッグ先の場領域がハイライト表示

4. ユーザーが場領域で離す
   - 場領域内で離した場合: カードが場に出る
   - 場領域外で離した場合: キャンセル（手札に戻る）

5. カード出し後の処理
   - 選択状態をリセット
   - サーバーに playCard リクエスト送信
   - ゲーム状態が更新
   - 画面が再描画
```

**複数枚選択の検証**

```
- 複数枚選択時は同じ数字のみ許可
- 異なる数字を選択した場合は前の選択を解除し、新しく選択されたカードを選択状態にする
- 場札の条件を満たさない場合はキャンセルで手札に戻る
```

#### 1.2 ドボン宣言ロジック

**常に表示されるドボン宣言UI**

```
1. ドボン宣言UIが常に画面に表示
   - 演算式選択ボタンが常に利用可能
   - ワンタップでドボン宣言可能

2. ユーザーが演算式を選択
   - 手札の全カードを使用する演算式のみ表示
   - 四則演算のうち1種類のみ
   - 複数の演算式がある場合は選択肢を表示

3. ユーザーが「宣言」ボタンをクリック
   - サーバーに declareDobo リクエスト送信
   - ゲーム状態が返し判定フェーズに移行

4. 返し判定フェーズ
   - ReturnDoboUI が表示
   - 各プレイヤーが返し宣言
   - 返し宣言は他プレイヤーに見えない
   - 返しが発生した場合は都度反映
```

**返しドボン宣言の表示**

```
- 返しが発生した場合、ゲーム画面に「返しドボン発生」と表示
- 返したプレイヤーの情報を表示
- 複数人の返しがある場合は最後に宣言したプレイヤーが勝者
```

#### 1.3 山札から引くロジック

```
1. ユーザーが「山札から引く」ボタンをクリック
2. サーバーに drawCard リクエスト送信
3. ゲーム状態が更新
4. 画面が再描画
5. ターンが次のプレイヤーに移行
```

### 2. ロビー・ルーム管理のロジック

#### 2.1 ルーム作成ロジック

```
1. ユーザーが「ルーム作成」ボタンをクリック
2. ルーム作成ダイアログが表示
3. ユーザーが基本レートを入力
4. 「作成」ボタンをクリック
5. サーバーに createRoom リクエスト送信
6. ルームが作成される
7. ルーム画面に遷移
```

#### 2.2 ルーム参加ロジック

```
1. ユーザーがロビー画面でルームを選択
2. 「参加」ボタンをクリック
3. サーバーに joinRoom リクエスト送信
4. ルームに参加
5. ルーム画面に遷移
```

#### 2.3 ゲーム開始ロジック

```
1. ルーム内の全プレイヤーが準備完了
2. ユーザーが「ゲーム開始」ボタンをクリック
3. サーバーに startGame リクエスト送信
4. ゲーム状態が返される
5. ゲーム画面に遷移
```

#### 2.4 無限ゲーム継続ロジック

```
1. ゲーム終了後、自動的に次のゲームが開始
2. ユーザーが「次で抜ける」を宣言
3. 次のゲーム終了時にプレイヤーが退出
4. 一人でも退出した場合、全員ルームに戻る
5. ルーム画面に遷移
```

### 3. プロフィール・統計画面のロジック

#### 3.1 プロフィール編集ロジック

```
1. ユーザーが「プロフィール編集」ボタンをクリック
2. プロフィール編集フォームが表示
3. ユーザーが情報を編集
4. 「保存」ボタンをクリック
5. サーバーに updateProfile リクエスト送信
6. プロフィールが更新
7. 画面が再描画
```

#### 3.2 統計情報表示ロジック

```
1. ユーザーが「統計」タブをクリック
2. サーバーに getStatistics リクエスト送信
3. 統計情報が返される
4. 統計情報が表示
```

#### 3.3 ランキング表示ロジック

```
1. ユーザーが「ランキング」タブをクリック
2. サーバーに getRanking リクエスト送信
3. ランキング情報が返される
4. ランキングが表示
```

### 4. ゲストログインロジック

#### 4.1 新規ゲスト作成

```
1. ユーザーが「新規ゲスト」を選択
2. サーバーに createGuest リクエスト送信
3. 新規ゲストIDが生成される
4. ゲストIDがローカルストレージに保存
5. ロビー画面に遷移
```

#### 4.2 既存ゲスト再利用

```
1. ユーザーが「既存ゲストID入力」を選択
2. ゲストID入力フォームが表示
3. ユーザーがゲストIDを入力
4. 「ログイン」ボタンをクリック
5. サーバーに authenticateGuest リクエスト送信
6. ゲストが認証される
7. ロビー画面に遷移
```

#### 4.3 ゲストID保存

```
1. ゲストログイン後、ゲストIDをローカルストレージに保存
2. 次回アクセス時に自動的にゲストIDを読み込み
3. ユーザーが手動でゲストIDを入力することも可能
```

---

### 5. リアルタイム同期ロジック

#### 5.1 ゲーム状態の同期

```
1. WebSocket で game:state イベントを受信
2. ゲーム状態をストアに保存
3. GameBoard コンポーネントが再描画
4. 画面が更新
```

#### 5.2 ドボン宣言の同期

```
1. WebSocket で game:dobo イベントを受信
2. ドボン宣言情報をストアに保存
3. 返し判定フェーズに移行
4. ReturnDoboUI が表示
```

#### 5.3 返しドボン宣言の同期

```
1. WebSocket で game:return イベントを受信
2. 返しドボン宣言情報をストアに保存
3. 返しが発生した場合、ゲーム画面に「返しドボン発生」と表示
4. 複数人の返しがある場合は最後に宣言したプレイヤーが勝者
5. 支払い情報が表示
```

#### 5.4 支払い情報の同期

```
1. WebSocket で game:payment イベントを受信
2. 支払い情報をストアに保存
3. PaymentUI が表示
4. 山札からのランダムカード引きの演出
5. 支払い金額が表示
```

### 6. エラーハンドリングロジック

#### 6.1 API エラー

```
- ネットワークエラー: リトライボタンを表示
- バリデーションエラー: エラーメッセージを表示
- サーバーエラー: エラーメッセージを表示
```

#### 6.2 ユーザーエラー

```
- 無効なカード操作: 警告メッセージを表示
- 無効なドボン宣言: 警告メッセージを表示
- 無効なゲストID: エラーメッセージを表示
```

#### 6.3 接続エラー

```
- WebSocket 接続切断: 再接続を試みる
- 再接続失敗: ユーザーに通知
```

### 7. 状態遷移ロジック

#### 7.1 ゲーム画面の状態遷移

```
playing
  ↓ (ドボン宣言)
dobo-declaration
  ↓ (返し判定)
return-dobo
  ↓ (支払い)
payment
  ↓ (ゲーム終了)
ended
  ↓ (次のゲーム開始)
playing
```

#### 7.2 ルーム画面の状態遷移

```
waiting
  ↓ (ゲーム開始)
playing
  ↓ (ゲーム終了)
waiting (無限ゲーム継続)
  ↓ (プレイヤー退出)
ended
```

### 8. パフォーマンス最適化ロジック

#### 8.1 レンダリング最適化

```
- 不要な再レンダリングを避ける
- メモ化を使用して計算結果をキャッシュ
- 仮想スクロールを使用してリスト表示を最適化
```

#### 8.2 メモリ最適化

```
- 不要なデータをメモリから削除
- 大きなデータ構造を避ける
- ガベージコレクションを活用
```

#### 8.3 ネットワーク最適化

```
- API リクエストをバッチ処理
- 不要なリクエストを避ける
- キャッシングを活用
```

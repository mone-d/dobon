## 2. デプロイメント環境

### 2.1 開発環境

**構成**:
- **フロントエンド**: ローカル（npm run dev）
- **バックエンド**: ローカル（npm run dev）
- **データベース**: Firebase（開発プロジェクト）
- **通信**: HTTP/WS（ローカルホスト）

**セットアップ**:
```bash
# フロントエンド
npm install
npm run dev

# バックエンド
npm install
npm run dev
```

**環境変数** (`.env.development`):
```
VITE_API_BASE_URL=http://localhost:3000
VITE_WS_URL=ws://localhost:3000
VITE_FIREBASE_API_KEY=dev_key
VITE_FIREBASE_AUTH_DOMAIN=dev.firebaseapp.com
VITE_FIREBASE_DATABASE_URL=https://dev.firebaseio.com
VITE_FIREBASE_PROJECT_ID=dev-project
```

### 2.2 本番環境

**構成**:
- **フロントエンド**: Vercel
- **バックエンド**: AWS EC2 または Heroku
- **データベース**: Firebase（本番プロジェクト）
- **通信**: HTTPS/WSS

**デプロイメント**:
```bash
# GitHub にプッシュ
git push origin main

# Vercel が自動的にデプロイ
# 1. ビルド: npm run build
# 2. デプロイ: dist/ をアップロード
# 3. CDN キャッシュ更新
```

**環境変数** (`.env.production`):
```
VITE_API_BASE_URL=https://api.dobon.example.com
VITE_WS_URL=wss://api.dobon.example.com
VITE_FIREBASE_API_KEY=prod_key
VITE_FIREBASE_AUTH_DOMAIN=prod.firebaseapp.com
VITE_FIREBASE_DATABASE_URL=https://prod.firebaseio.com
VITE_FIREBASE_PROJECT_ID=prod-project
```

### 4.2 データフロー

**ゲーム状態更新フロー**:
```
1. ユーザーがカードを出す
   ↓
2. フロントエンドが REST API に POST
   ↓
3. バックエンドが Firebase に書き込み
   ↓
4. Firebase が WebSocket で全クライアントに通知
   ↓
5. フロントエンドが画面を更新
```

## 5. スケーラビリティ

### 5.1 フロントエンドのスケーリング

**Vercel の自動スケーリング**:
- トラフィック増加時に自動的にスケール
- トラフィック減少時に自動的にスケールダウン
- 追加設定不要

**CDN キャッシング**:
- グローバル CDN から配信
- エッジロケーションでキャッシュ
- レイテンシー低減

### 5.2 バックエンドのスケーリング

**AWS EC2**:
- Auto Scaling Group で自動スケーリング
- ロードバランサーで負荷分散
- 複数インスタンスで冗長性確保

**Heroku**:
- Dyno の自動スケーリング
- ロードバランサー内蔵
- シンプルなスケーリング

### 5.3 データベースのスケーリング

**Firebase Realtime Database**:
- 自動スケーリング
- 追加設定不要
- 使用量に応じた課金

## 6. 可用性・復帰

### 6.1 フロントエンドの可用性

**Vercel**:
- 99.95% の可用性 SLA
- 複数リージョンでの冗長性
- 自動フェイルオーバー

### 6.2 バックエンドの可用性

**AWS EC2**:
- 複数 AZ での冗長性
- Auto Scaling Group で自動復帰
- ロードバランサーで自動フェイルオーバー

**Heroku**:
- 複数 Dyno での冗長性
- 自動復帰機能
- ロードバランサー内蔵

### 6.3 データベースの可用性

**Firebase Realtime Database**:
- 複数リージョンでの冗長性
- 自動バックアップ
- 99.95% の可用性 SLA

## 7. セキュリティ

### 7.1 通信セキュリティ

**HTTPS/WSS**:
- すべての通信を暗号化
- SSL/TLS 証明書（Let's Encrypt）
- 自動更新

### 7.2 認証・認可

**ゲストID**:
- ローカルストレージに保存
- JWT トークンで認証
- ステートレス設計

### 7.3 データセキュリティ

**Firebase セキュリティルール**:
```json
{
  "rules": {
    "gameHistory": {
      ".read": "auth != null",
      ".write": "auth != null"
    },
    "statistics": {
      ".read": "auth != null",
      ".write": "auth != null"
    }
  }
}
```

## 9. デプロイメント手順

### 9.1 初回デプロイメント

```bash
# 1. Vercel にサインアップ
# https://vercel.com

# 2. GitHub リポジトリを連携
# Vercel ダッシュボード → Import Project

# 3. 環境変数を設定
# Vercel ダッシュボード → Settings → Environment Variables

# 4. デプロイ
# GitHub に main ブランチをプッシュ

# 5. デプロイ完了
# Vercel が自動的にビルド・デプロイ
# https://dobon.vercel.app で利用可能
```

### 9.2 更新デプロイメント

```bash

git add .

# 4. Vercel が自動デプロイ
```

## 10. トラブルシューティング

### 10.1 デプロイメント失敗

**原因**: ビルドエラー

**対応**:
```bash
# ローカルでビルドテスト
npm run build

# エラーメッセージを確認
# Vercel ダッシュボード → Deployments → Failed

# エラーを修正してプッシュ
git push origin main
```

### 10.2 本番環境でのエラー

**原因**: 環境変数の設定ミス

**対応**:
```bash
# 環境変数を確認
# Vercel ダッシュボード → Settings → Environment Variables

# 環境変数を修正
# Vercel が自動的に再デプロイ
```

### 10.3 パフォーマンス低下

**原因**: キャッシング設定の問題

**対応**:
```bash
# キャッシング設定を確認
# vercel.json を確認

# CDN キャッシュをクリア
# Vercel ダッシュボード → Deployments → Redeploy
```

# フロントエンド・バックエンド統合テスト計画

## テスト目的
Unit 1 (Frontend) と Unit 2 (Backend) の統合を確認し、WebSocket/API通信が正常に動作することを検証する。

## テスト環境
- **フロントエンド**: http://localhost:5173/
- **バックエンド**: http://localhost:3000/
- **WebSocket**: ws://localhost:3000/

## 前提条件
- [x] バックエンドサーバーが起動している (Port 3000)
- [x] フロントエンドサーバーが起動している (Port 5173)
- [x] 環境変数が正しく設定されている

---

## Phase 1: 基本接続テスト

### 1.1 ヘルスチェック
- [ ] GET /health が200を返す
- [ ] レスポンスに status: 'ok' が含まれる

### 1.2 WebSocket接続
- [ ] フロントエンドからWebSocket接続が確立される
- [ ] 接続時に 'connection' イベントがバックエンドで受信される
- [ ] 接続IDが正しく管理される

---

## Phase 2: ルーム管理テスト

### 2.1 ルーム作成
- [ ] フロントエンドから 'room:create' イベントを送信
- [ ] バックエンドでルームが作成される
- [ ] ルームIDとルームコードが返される
- [ ] 作成者がルームに参加する

### 2.2 ルーム参加
- [ ] フロントエンドから 'room:join' イベントを送信
- [ ] バックエンドでプレイヤーがルームに追加される
- [ ] ルーム内の全プレイヤーに 'room:updated' イベントが送信される
- [ ] プレイヤー一覧が更新される

### 2.3 ルーム退出
- [ ] フロントエンドから 'room:leave' イベントを送信
- [ ] バックエンドでプレイヤーがルームから削除される
- [ ] ルーム内の全プレイヤーに 'room:player-left' イベントが送信される

---

## Phase 3: ゲーム開始テスト

### 3.1 ゲーム開始
- [ ] フロントエンドから 'game:start' イベントを送信
- [ ] バックエンドでゲームセッションが作成される
- [ ] 初期手札が配られる
- [ ] 場札が設定される
- [ ] 全プレイヤーに 'game:started' イベントが送信される
- [ ] 各プレイヤーに 'game:state-updated' イベントが送信される

### 3.2 初期状態確認
- [ ] 各プレイヤーの手札枚数が正しい
- [ ] 場札が表示される
- [ ] 現在のターンプレイヤーが表示される
- [ ] 倍率が表示される（初期Aの枚数に応じて）

---

## Phase 4: カード操作テスト

### 4.1 カードプレイ
- [ ] フロントエンドから 'game:play-card' イベントを送信
- [ ] バックエンドでカードが検証される
- [ ] 有効なカードの場合、場札が更新される
- [ ] 全プレイヤーに 'game:card-played' イベントが送信される
- [ ] 全プレイヤーに 'game:state-updated' イベントが送信される
- [ ] 次のプレイヤーにターンが移る

### 4.2 カードドロー
- [ ] フロントエンドから 'game:draw-card' イベントを送信
- [ ] バックエンドで山札からカードが引かれる
- [ ] 引いたカードが手札に追加される
- [ ] 全プレイヤーに 'game:card-drawn' イベントが送信される
- [ ] 引いたプレイヤーのみカード情報が見える

### 4.3 スート選択（8のワイルド）
- [ ] 8を出した後、スート選択画面が表示される
- [ ] フロントエンドから 'game:select-suit' イベントを送信
- [ ] バックエンドでスートが設定される
- [ ] 全プレイヤーに 'game:suit-selected' イベントが送信される

## Phase 5: 特殊カードテスト

### 5.1 2のカード（強制ドロー）
- [ ] 2を出すと次のプレイヤーが2枚引く
- [ ] 2を重ねると枚数がスタックされる（2+2=4枚）
- [ ] 強制ドロー後、ターンがスキップされる

### 5.2 8のカード（スキップ）
- [ ] 8を出すと次のプレイヤーがスキップされる
- [ ] スキップされたプレイヤーの次にターンが移る

### 5.3 Jのカード（リバース）
- [ ] Jを出すとターン順が逆転する
- [ ] 逆転後、正しい順序でターンが進む

### 5.4 Kのカード（手札公開）
- [ ] Kを出すと次のプレイヤーの手札が公開される
- [ ] 公開された手札が全プレイヤーに表示される
- [ ] 次のカードが出されるまで公開状態が続く

## Phase 6: ドボン宣言テスト

### 6.1 ドボン宣言
- [ ] フロントエンドから 'game:declare-dobo' イベントを送信
- [ ] バックエンドでドボン式が検証される
- [ ] 有効な式の場合、ドボンフェーズに移行する
- [ ] 全プレイヤーに 'game:dobo' イベントが送信される
- [ ] 倍率が2倍される
- [ ] 10秒のタイマーが開始される

### 6.2 返しドボン宣言
- [ ] フロントエンドから 'game:declare-return' イベントを送信
- [ ] バックエンドで返しドボン式が検証される
- [ ] 有効な式の場合、返しドボンが成功する
- [ ] 全プレイヤーに 'game:return' イベントが送信される
- [ ] 倍率が2倍される
- [ ] 最後の返しドボンプレイヤーが勝者となる

### 6.3 返さない宣言
- [ ] フロントエンドから 'game:declare-no-return' イベントを送信
- [ ] バックエンドで返さない宣言が記録される
- [ ] 他プレイヤーには通知されない（非表示ルール）
- [ ] 全員が返さない場合、最初のドボン宣言者が勝者となる

### 6.4 無効なドボン宣言
- [ ] 無効な式でドボン宣言する
- [ ] バックエンドでペナルティが発生する
- [ ] 全プレイヤーに 'game:penalty' イベントが送信される
- [ ] ペナルティプレイヤーがバーストする

# ドボンゲーム - 要件ドキュメント

## 1. プロジェクト概要

### 1.1 プロジェクト名
**ドボン - オンライン対戦カードゲーム**

### 1.2 プロジェクトタイプ
- **分類**: 新規開発（Greenfield）
- **スコープ**: 中規模
- **複雑度**: 中程度
- **開発期間**: 1-2週間（MVP）

### 1.3 プロジェクト概要
チームメイト向けのオンライン対戦カードゲーム「ドボン」のシステム化。オフラインで遊んでいたオリジナルルールをデジタル化し、リアルタイムでのオンライン対戦を実現する。POCレベルの実装で、チームメイト（5-10人程度）のみを対象とする。

### 1.4 ユーザー
- **対象ユーザー**: 開発者のチームメイト（5-10人程度）
- **ユーザー数**: 小規模（スケーラビリティは不要）
- **利用環境**: モバイルデバイス（iOS/Android）のWebブラウザ
- **認証**: ゲストログイン（将来的にLINE連携予定）

## 2. 機能要件

### 2.1 ゲームプレイ機能

#### 2.1.1 基本ゲームフロー
- **プレイヤー数**: 2-8人対戦に対応
- **ゲーム進行**: ターンベース（ただしドボン宣言にはリアルタイム性が必要）
- **勝利条件**: ドボン宣言による上がりのみ

#### 2.1.2 カード操作
- 場札に対して条件を満たすカードを出す（同じ数字またはスート）
- 複数枚の同数字カードをまとめて出す
- 山札から1枚引く（任意、1ターンに1枚のみ）

#### 2.1.3 特殊カード効果
- **A(1)**: 次プレイヤーのターンをスキップ
- **2**: 次プレイヤーが山札から2枚引く
- **8**: ワイルド（任意のカードに対して出せる、スート指定可能）
- **J(11)**: ターン順を逆回りにする
- **K(13)**: 次プレイヤーの手札を全員に公開（そのプレイヤーの次ターン終了まで）

#### 2.1.4 ドボン宣言
- **条件**: 現在の手札の全カードを使用し、四則演算（+, -, *, /）のうち1種類のみで場札の数字を作る
- **制約**: 自分が直前に出した場札に対してはドボン不可
- **タイミング**: 
  - 自分の出したカードが場の最新でない時はいつでもドボン可能
  - つまり、自分のターン中（相手のカードが場にある状態）でも、相手のターン中でも宣言可能
- **UI**: ユーザーが手札を全て使用する演算式を選択する形式
- **ペナルティ**: 演算式が場のカードと一致しない場合、バースト時と同じ負け金を支払う（敗北）

#### 2.1.4.1 ドボン宣言後の返し判定フロー
- **ドボン宣言時**: 誰かがドボンを宣言すると、ゲームは一時停止
- **返し判定フェーズ**: 全プレイヤーに対して「返しドボンを宣言するか」を個別に確認
  - 各プレイヤーの返し宣言は他のプレイヤーには見えない（ノーリスク返しを防ぐため）
  - 全プレイヤーが返し宣言の有無を確定させるまで待機
- **返し判定結果**:
  - 返しがない場合: 元のドボン宣言者が勝者、ゲーム終了
  - 返しがある場合: 複数人がドボン可能な場合、最後に宣言したプレイヤーが勝者、ゲーム終了

#### 2.1.5 バースト
- **条件**: 手札が14枚になった時点で即敗北
- **支払い**: 敗者は山札から1枚引き、その数値に倍率を掛けた金額を自分以外の全プレイヤーに支払う

#### 2.1.7 倍率システム
倍率は以下の条件を掛け合わせて計算される（total_multiplier = 2^n）：

- **初期A**: 開始時にAが連続した回数分（×2 per occurrence）
- **引きドボン**: 山札から引いた結果ドボン成立（×2）
- **オープンドボン**: K(13)による公開状態でドボン（×2）
- **返しドボン**: 返し回数ごとに累積（×2 per counter）
- **山札再生成**: 山札切れ時に捨て札をシャッフルして再生成（×2 per reshuffle）

#### 2.1.8 支払い
- **通常ドボン**: 勝者に対して、上がられた場札を出したプレイヤーが支払う
- **支払い額**: 山札からランダムに引いた1枚の数値 × 基本レート × 倍率
- **バースト支払い**: 敗者が自分以外の全プレイヤーに同額支払い

#### 2.1.9 山札管理
- 山札切れ時、場の最初の1枚を除き捨て札をシャッフルして再生成
- 再生成ごとに倍率が×2される

### 2.2 ゲームロビー・マッチング

#### 2.2.1 ルーム管理
- プレイヤーが手動でルーム作成・参加
- ルームコード（短いID）でルーム識別
- ルーム内でプレイヤーが基本レート（掛け金）を合意して設定

#### 2.2.2 ゲーム開始
- ルーム内の全プレイヤーが準備完了後、ゲーム開始
- 最小2人、最大8人

### 2.3 ユーザー管理

#### 2.3.1 認証
- **ゲストログイン**: 
  - 新規ゲストIDを生成するか、既存のゲストIDを入力して再利用するかを選択可能
  - ゲストIDはブラウザのローカルストレージに保存可能（ユーザーの選択）
- **将来対応**: LINE連携（MVP では実装不要）

#### 2.3.2 ユーザープロフィール
- ユーザー名
- アバター画像
- 自己紹介

### 2.4 ゲーム履歴・統計

#### 2.4.1 ゲーム履歴
- 過去のゲーム結果を保存・表示
- ゲーム日時、参加プレイヤー、結果、獲得/支払い金額を記録

#### 2.4.2 統計情報
- 総ゲーム数
- 勝利数
- 敗北数
- 勝率
- 総獲得金額
- 総支払金額
- 平均獲得金額
- 最高倍率

#### 2.4.3 ランキング
- **総獲得金額ランキング**: チーム内での獲得金額ランキング表示

## 3. 非機能要件

### 3.1 パフォーマンス
- **優先度**: 安定性重視
- **遅延許容度**: ドボン宣言のタイミングが重要なため、極力遅延は許容しない
- **リアルタイム通信**: WebSocket推奨（ポーリングでも可）
- **接続復帰**: 接続が切れた場合、ゲーム停止で対応

### 3.2 スケーラビリティ
- **対象ユーザー**: 5-10人程度（チームメイトのみ）
- **大規模化**: 想定していない
- **サーバー負荷**: 低い

### 3.3 セキュリティ
- **要件**: POCレベル、高いセキュリティ要件なし
- **認証**: ゲストログイン（簡易的）
- **通信**: HTTPS推奨

### 3.4 ブラウザ対応
- **対応範囲**: 最新ブラウザのみ対応
- **プラットフォーム**: iOS Safari、Android Chrome

### 3.5 UI/UX
- **デザイン方針**: リアルなカードゲーム風
- **レスポンシブ**: モバイルデバイス対応（スマートフォン）
- **ダークモード**: 不要
- **デザイン重視**: ドボン宣言時のUI、最後の掛け金を決めるための山札からのランダムカード引きの演出に力を入れる

## 4. 技術要件

### 4.1 推奨技術スタック

#### バックエンド
- **言語**: Node.js（TypeScript推奨）
- **フレームワーク**: Express.js または Nest.js
- **リアルタイム通信**: Socket.io（WebSocket）
- **データベース**: PostgreSQL または MongoDB
- **認証**: JWT（簡易的）

#### フロントエンド
- **言語**: TypeScript
- **フレームワーク**: Vue.js 3 または React
- **UI/UXライブラリ**: Tailwind CSS または Material-UI
- **リアルタイム通信**: Socket.io クライアント
- **状態管理**: Pinia（Vue）または Redux（React）

### 4.2 デプロイ環境
- **決定**: 後日決定（AWS、GCP、Azure、自社サーバーなど）
- **当面**: ローカル開発環境で動作確認

### 4.3 開発環境
- **バージョン管理**: Git
- **パッケージ管理**: npm または yarn
- **ビルドツール**: Vite または Webpack

## 5. MVP スコープ

### 5.1 MVP に含まれる機能
1. **ゲームプレイ**: 基本的なゲーム進行、カード操作、ドボン宣言
2. **ユーザー管理**: ゲストログイン、プロフィール管理
3. **ゲーム履歴**: ゲーム結果の保存・表示
4. **統計情報**: 基本的な統計情報の表示
5. **ランキング**: 総獲得金額ランキング

### 5.2 MVP に含まれない機能
- LINE連携認証
- リプレイ機能
- チャット機能
- 高度なセキュリティ機能

---

## 6. 制約・注意事項

### 6.1 ゲームロジック
- ドボン宣言の演算式検証は厳密に実装（ペナルティ対象）
- 倍率計算は複雑なため、テストケースを充実させる
- 山札再生成ロジックの正確な実装が必要

### 6.2 UI/UX
- ドボン宣言時のUI は直感的で使いやすい設計が必須
- 最後の掛け金を決めるための山札からのランダムカード引きの演出は、ワクワク感を演出する設計が必須
- モバイルデバイスでの操作性を重視

### 6.3 リアルタイム通信
- ゲーム中の状態同期は確実に行う必要がある
- 接続が切れた場合の復帰ロジックを実装

### 6.4 開発期間
- 1-2週間の短期開発のため、スコープ管理が重要
- 複雑な機能は段階的に実装

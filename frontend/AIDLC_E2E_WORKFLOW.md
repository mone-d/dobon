# AIDLC E2Eテスト統合ワークフロー

## 概要

今後のAIDLC開発では、すべての機能実装後にE2Eテストを実行し、動作確認を自動化します。

## ワークフロー

### 1. 機能実装
- 新機能の実装
- コンポーネントの作成・修正

### 2. E2Eテスト実行
```bash
npm run test:e2e
```

### 3. テスト結果の確認
- ✅ すべてのテストが成功 → 次のステップへ
- ❌ テストが失敗 → 修正して再実行

### 4. 失敗時の対応

#### スクリーンショットを確認
```bash
open test-results/
```

#### エラーコンテキストを確認
```bash
cat test-results/*/error-context.md
```

#### テストまたは実装を修正
- UIテキストの変更 → テストを修正
- 機能の不具合 → 実装を修正

#### 再実行
```bash
npm run test:e2e
```

### 5. コミット
すべてのテストが成功したら：
```bash
git add .
git commit -m "feat: <機能名>の実装とE2Eテスト"
```

## テストコマンド

### 基本実行
```bash
npm run test:e2e
```

### UIモード（デバッグ用）
```bash
npm run test:e2e:ui
```

### レポート表示
```bash
npm run test:e2e:report
```

## 現在のテストカバレッジ

### ✅ 実装済みテスト

1. **基本フロー**
   - ゲストログイン
   - ロビー画面への遷移
   - デバッグナビゲーションでゲーム画面へ
   - ゲーム画面の要素確認

2. **WebSocket接続確認**
   - 接続ログの確認

3. **エラー通知の表示**
   - エラーハンドリングの確認

4. **デバッグナビゲーション**
   - 画面遷移の確認

### 📝 今後追加すべきテスト

1. **ルーム作成・参加フロー**
   - ルーム作成
   - ルーム参加
   - プレイヤー準備状態

2. **ゲームプレイ**
   - カード選択
   - カードプレイ
   - カードドロー
   - ドボン宣言

3. **エラーケース**
   - バックエンド停止時の挙動

   - 無効な操作の確認

## ベストプラクティス

### テスト作成時
- 実際のUIテキストを使用する
- タイムアウトを適切に設定する（5秒〜10秒）
- スクリーンショットを活用する

### テスト実行時
- バックエンドが起動していることを確認
- フロントエンドの開発サーバーは自動起動される
- 失敗時はスクリーンショットを確認

### テスト保守時
- UIが変更されたらテストも更新
- 新機能追加時は対応するテストも追加
- 定期的にテストを実行してリグレッションを防ぐ
